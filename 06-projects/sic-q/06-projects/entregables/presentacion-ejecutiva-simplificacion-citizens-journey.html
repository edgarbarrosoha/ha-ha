01 / reporte de simplificación

Simplificar para lanzar el SIC-Q.

El citizens journey v03 sirvió para comunicar visión. Este deck lo traduce a una versión ejecutable para piloto: menos fases, menos fricción, más claridad institucional.

DECISIÓN / qué simplificar ahora
01
02 / diagnóstico

La versión actual explica el sistema completo, pero mezcla la experiencia ciudadana con la maquinaria técnica.

PROBLEMA 01

Journey y backend aparecen como si fueran lo mismo.

Las fases de base de conocimiento, sensemaking, agente institucional y gobierno no son pasos que vive la persona usuaria.

PROBLEMA 02

Hay más opciones de las que el ciudadano necesita.

Cuatro tipos de participación, once fases y varias decisiones tempranas elevan fricción cognitiva antes de generar valor.

PROBLEMA 03

Se pide demasiado antes de dejar participar.

El registro con CURP, correo, celular y verificación oficial bloquea adopción antes de que la plataforma entregue una primera experiencia útil.

02
03 / complejidad visible

La complejidad del diseño actual se concentra en cuatro puntos de fricción.

11 FASES

Demasiadas transiciones para una interacción que debería sentirse continua.

4 JOURNEYS

Dos de ellos son redundantes o deberían existir como capacidades transversales.

6+ DATOS DE REGISTRO

La exigencia de identidad completa aparece demasiado pronto, incluso para apoyar una idea.

2 BASES

La separación técnica se comunica como si fueran dos sistemas distintos frente al usuario.

03
04 / principio rector

Lo que el ciudadano vive va en el journey. Lo que el sistema procesa va en arquitectura.

Ese corte permite documentar mejor, diseñar mejor y construir un MVP sin inflar la interfaz con pasos que nadie percibe.

04
05 / nuevo journey

El citizens journey puede pasar de once fases a cinco momentos comprensibles.

ANTES / v03

Diseño demasiado secuencial.

Diseño participativo → Descubrimiento → Primer contacto → Invitación → Orientación → Participación → Base de conocimiento → Sensemaking → Agente institucional → Gobierno → Feedback → Ciclo virtuoso.

DESPUÉS / v04

Recorrido claro para piloto.

Co-diseño → Descubrimiento → Participar → Seguimiento → Respuesta. El backend queda documentado aparte como almacenamiento, análisis y canalización.

01

Co-diseño

Construir con ciudadanía antes de escalar.

02

Descubrimiento

WhatsApp, web, kiosco o taller.

03

Participar

Llega, decide y actúa sin pasos artificiales.

04

Seguimiento

Ranking, progreso y tiempo restante.

05

Respuesta

Resultado institucional e invitación a continuar.

05
06 / participación

La oferta de participación se reduce a dos acciones reales y una capacidad transversal del agente.

ACCIÓN 01

Proponer

Crear una idea nueva mediante diálogo guiado. Requiere registro verificado porque inicia una propuesta que puede escalar a deliberación.

ACCIÓN 02

Apoyar

Votar una propuesta y, si hace sentido, compartirla. Sustituye “impulsar” y absorbe el journey de “transformar”.

CAPACIDAD

Platicar

Siempre disponible. No es una categoría aparte; es la forma natural en la que el agente acompaña y detecta intención.

El menú de cuatro cards desaparece. La interfaz ideal es más simple: Tengo una idea, Quiero ver ideas de otros o texto libre.

06
07 / identidad y registro

El cambio de mayor impacto es pasar de registro total upfront a registro progresivo.

NIVEL 01

Visitante

Explora propuestas, consulta datos y conversa con el agente sin entregar información personal.

REQUISITOS / ninguno
NIVEL 02

Participante

Apoya ideas y recibe seguimiento con una fricción mínima orientada a celular y código postal.

REQUISITOS / celular + código postal
NIVEL 03

Ciudadano verificado

Puede publicar propuestas. La verificación fuerte se reserva para el momento en que una idea cruza un umbral institucional.

REQUISITOS / celular + CURP + código postal

La implicación ejecutiva es directa: eliminar INE como requisito universal y dejarla como validación opcional o tardía cambia la tasa esperada de adopción.

07
08 / simplificación del piloto

El piloto también debe reducir complejidad operacional, no solo complejidad de interfaz.

SELECCIÓN DE PROPUESTAS

Menos reglas para salir antes.

De tres ciclos al año y tres vías de selección a un solo ciclo de 4-6 meses, con top N por apoyos y 20-30 propuestas a deliberación.

CO-DISEÑO

Menos actividades paralelas.

De cuatro actividades en nueve semanas a tres actividades en seis semanas: dos talleres, 300-500 conversaciones IA y una mesa técnica unificada.

1 CICLO PILOTO

Menor complejidad de gobernanza y comunicación.

20-30 PROPUESTAS

Volumen compatible con capacidad real de deliberación.

6 SEMANAS

Plazo más realista para validar antes de desarrollar demasiado.

0 GAMIFICACIÓN

Puntos y rachas salen del piloto para no trivializar participación.

08
09 / ajustes de producto

Tres cambios de experiencia mejoran el flujo sin elevar costo de desarrollo.

INTERFAZ

Dos opciones y texto libre.

Desaparece el menú con cuatro journeys. El agente interpreta intención en vez de forzar categorización previa.

DUPLICADOS

Información lateral, no bloqueo.

La propuesta se publica y luego se invita a unir fuerzas con ideas parecidas. Se protege el momentum de participación.

ARQUITECTURA

Una base, dos capas lógicas.

Ciudadanía y contexto viven en una misma base de datos con vistas distintas. Se simplifica mantenimiento y análisis.

09
10 / prioridades y decisiones

Cinco decisiones destraban casi todo el rediseño.

ALTA PRIORIDAD

Separar journey y backend.

Clarifica producto, narrativa y alcance técnico desde el inicio.

ALTA PRIORIDAD

Confirmar si INE es obligación legal o costumbre.

La respuesta define si el registro progresivo es viable.

ALTA PRIORIDAD

Definir cuántas propuestas puede absorber gobierno.

Ese número fija umbral, ranking y capacidad de respuesta.

ALTA PRIORIDAD

Escoger canal del MVP.

Solo WhatsApp reduce complejidad. Web y kioscos amplían inclusión, pero elevan el alcance inicial.

ALTA PRIORIDAD

Definir si co-diseño va antes o en paralelo.

Afecta calendario, presupuesto y qué se aprende antes de construir.

ENFOQUE

Lanzar simple, aprender rápido.

El criterio rector del piloto debe ser confianza ciudadana y capacidad operativa, no completitud.

10
11 / lo que se conserva

Simplificar no significa abandonar lo que sí le da valor al SIC-Q.

Agente conversacional como interfaz.

Sigue siendo el diferenciador principal entre ciudadanía, gobierno y datos.

Diálogo guiado para estructurar propuestas.

La IA sigue reduciendo barrera de entrada y ordenando lenguaje ciudadano.

Clustering semántico y análisis HA.

El valor técnico real permanece, solo deja de mostrarse como fase visible del usuario.

Feedback y rankings transparentes.

La promesa central se mantiene: la voz ciudadana no desaparece en un hoyo negro.

Canales múltiples a futuro.

WhatsApp, web y kioscos siguen siendo compatibles con inclusión territorial.

Instituto del Futuro como garantía.

La continuidad transexenal sigue siendo una pieza política e institucional clave.

11
12 / cierre

La recomendación es simple: lanzar un sistema más pequeño, más legible y más defendible.

Si el piloto quiere construir confianza, la experiencia debe ser sencilla para la ciudadanía y operable para gobierno. La simplificación propuesta no reduce ambición: ordena prioridades.

Siguiente secuencia recomendada: validar internamente, producir el citizens journey v04, separar la arquitectura backend y cerrar la decisión jurídica sobre verificación.
